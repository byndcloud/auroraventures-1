# Authentication & sessions

Authentication answers "who is calling." Bugs here lead to **account takeover** — the most expensive class of bug after data leaks.

For authorization (what they can do), see `access-control.md`. The two are independent and need to be checked separately.

## Password handling

### Storage

- **Hash with `argon2id`** (preferred) or **`bcrypt`** with a current work factor. Never `md5`, `sha1`, or unsalted hashes — even `sha256` alone is broken for passwords (too fast to brute-force).
- **Never** encrypt passwords reversibly. There is no use case where you need the plaintext later; the system is designed not to.
- If you find passwords stored in plaintext, base64, or with reversible encryption: critical finding. Recommend immediate hash migration plus user-forced-reset.

### Login flow

- **Constant-time response** for "user not found" vs "wrong password" — same status, same timing, same message. Otherwise the login endpoint is also a user-enumeration endpoint.
- **Rate-limit per account and per IP.** Failures only is fine; some implementations also throttle successes from one IP for slow-roll account spraying.
- **Lockout strategies are tradeoffs.** Hard lockout enables denial-of-service against accounts. Prefer rate limiting + step-up auth (CAPTCHA, MFA challenge) on suspicious patterns.
- **Successful login should rotate the session ID** to prevent session fixation.

### Password reset

The most common takeover vector. Check:

- **Token is generated by a CSPRNG.** Not `Math.random()`. Not `Date.now()`. Not the user's email + a salt.
- **Token is one-time.** Used once, invalidated. Especially after the password is changed.
- **Token is short-lived.** Minutes to ~1 hour. Long-lived reset tokens that sit in inboxes are bad.
- **Token is bound to the account.** Sending the token cookie to a different account on click should fail.
- **Successful reset invalidates all existing sessions and refresh tokens.** Otherwise an attacker who already has a session keeps it after the user "resets."
- **Same response for "account exists" vs "doesn't exist"** on the request side — otherwise it's a user enumeration endpoint. ("If your email is registered, we sent you a link.")

## Session management

### Cookies

For browser-facing apps, cookies are the right session carrier:
- `HttpOnly` — JS can't read it (defense vs XSS stealing the session).
- `Secure` — HTTPS-only.
- `SameSite=Lax` (default) or `Strict` (more conservative). `None` only if you genuinely need cross-site (and is paired with `Secure`).
- A cookie name that doesn't reveal the framework (e.g., not the default `connect.sid`).
- A reasonable lifetime — short for sensitive apps, long for low-risk UX, but always with server-side revocation possible.

If you see a session cookie missing `HttpOnly` or `Secure` in production code, finding.

### Session rotation

The session ID must be regenerated on:
- **Login** — prevents session fixation (attacker pre-sets a known session ID, victim logs in, attacker now has the authenticated session).
- **Privilege escalation** — if a user assumes admin or switches roles.
- **Logout** — invalidate server-side, not just client-side.
- **Password change.** All sessions, all devices.

A "logout" that only deletes the client cookie without invalidating server-side state is a bug.

### Storage

Where is the session-validity check made?
- **Server-side store** (Redis, DB) — strongest; server can revoke individually and globally.
- **Signed cookie with no server check** — fast, stateless, but **cannot be revoked** until expiry. Don't use for high-value sessions; pair with short TTLs and refresh tokens that *are* revocable.
- **JWT only** — same problem as signed cookies, with extra footguns (see below).

## JWT specifics

JWTs are useful, easy to misuse. Specific things to check:

- **Algorithm is pinned.** Verifier specifies the expected algorithm; doesn't trust whatever the token's `alg` header claims. The `none` algorithm attack is decades old and still found in the wild.
- **`alg` is asymmetric** (`RS256`, `EdDSA`) when issuer and verifier are different services. `HS256` only when they're the same trust boundary; otherwise, sharing the symmetric key broadly is a footgun.
- **`aud`, `iss`, `exp`, `nbf` are validated.** A token meant for service A is rejected by service B (which validates `aud`).
- **Tokens are short-lived.** Minutes to ~1 hour, paired with refresh tokens that are revocable.
- **Authorization is re-checked from current state**, not from cached JWT claims. Roles change; tokens linger.
- **No PII or secrets in claims.** Base64, not encryption.
- **Signing key is rotatable.** A `kid` field + a JWKS endpoint (or equivalent) lets you rotate without downtime.
- **Refresh tokens are stored server-side** and are revocable. A "refresh token" that's just a long-lived JWT is not a refresh token.

In code review, if you see manual JWT decoding (`base64decode(token.split('.')[1])`), trace whether signature verification happens **before** that. Decoding without verifying is a frequent bug; the attacker controls the bytes you're now reading.

## Multi-factor authentication

For accounts with elevated access (admins, billing, anything that touches money or others' data), MFA should not be optional:

- **TOTP** is the minimum bar.
- **WebAuthn / passkeys** are stronger and worth adopting where supported.
- **SMS-based MFA** is weaker than nothing in some threat models (SIM swapping); avoid for high-value accounts.

Things to check:
- Is MFA required for the right operations? "Step-up" authentication requires MFA again for sensitive actions even mid-session (changing password, adding a payment method, exporting data).
- Are MFA recovery codes one-time, time-limited, and stored hashed?
- Can MFA be removed without re-authentication? Should not be possible.
- Is there a backup recovery mechanism that effectively bypasses MFA? (Email-based recovery → MFA strength is bounded by email account security.) Document the tradeoff.

## OAuth / OIDC

If the codebase implements an OAuth client or provider:

- **`state` parameter** is required for the authorization code flow — without it, CSRF on the callback is possible.
- **PKCE** for public clients (mobile/SPA). For confidential clients it's recommended too.
- **Redirect URI is allowlisted exactly** — no wildcards, no path-prefix matches that allow attacker-controlled paths.
- **Scopes are minimized.** Don't request scopes you don't use.
- **The `code` is exchanged server-side** — never leak the client secret to the browser.
- **Token endpoint validates the redirect_uri** matches the one used in the authorization request.

Implementing OAuth from scratch is a footgun farm; prefer a vetted library.

## API keys (server-to-server)

- **Long, random, opaque.** No structure that gives away the issuer.
- **Scoped.** Per-environment, per-tenant, per-purpose. A single key that can do anything is a single key that, when leaked, does anything.
- **Rotatable** without downtime.
- **Hashed at rest.** When a key is created, show it once; store only the hash. (Like passwords.) "Look up the customer's API key from the support panel" is a smell.
- **Auditable.** Every key has a created-by, last-used, last-rotated.

## Account takeover surfaces beyond login

Easy-to-miss takeover paths:

- **Account merge / link.** Adding an OAuth provider to an existing account: the link should require re-auth, and shouldn't trust the email of the OAuth provider as proof of ownership of an existing account with that email.
- **Email change.** Should require old password + verification of the new email. Otherwise, a stolen session changes the email and locks the real owner out.
- **Phone change.** Same.
- **MFA reset.** Resetting MFA via email is only as secure as the email account.
- **API key creation.** A stolen session can create a long-lived API key that survives session expiry / password reset (unless those events also revoke keys).
- **Trusted-device management.** "Remember this device" features should be visible to the user and revocable.

## Anonymous / public endpoints

Endpoints that don't require auth at all are a category, not an oversight. Check:

- Is this endpoint **intentionally** public? (Documented, reviewed.)
- Does it have **rate limiting**? Public endpoints are free fuel for abuse.
- Does it leak more than it needs? (User-enumeration via signup, email-existence via login, etc.)
- If it triggers email/SMS, is there a per-recipient cap to prevent abuse-as-spam-cannon?

## What to call out, what to skip

Worth raising:
- Plaintext / md5 / sha1 / unsalted password storage.
- Reset tokens generated with non-CSPRNG.
- Reset tokens that don't expire / aren't single-use.
- Sessions that aren't rotated on login.
- JWT verifier that trusts the token's `alg`.
- JWT used as a long-lived session with no revocation path on a sensitive surface.
- MFA absent on admin / billing flows.
- A login endpoint with different responses for "user exists" vs "wrong password."

Not worth raising:
- "MFA is optional for end users." Often correct UX; flag only if context demands stronger.
- "Session lifetime is 30 days." Depends on app risk profile.
- "Cookies don't have `Secure` in dev config." Check it's `Secure` in prod config.
